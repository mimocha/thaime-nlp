# Research Plan 003: Component-Level Romanization Framework

**Date:** 2026-03-06
**Branch:** `research/003-component-romanization`
**Status:** Planned
**Prerequisites:** Research 001 (romanization source audit), Research 002 (variant generator)

## Research Question

Can Thai informal romanization be decomposed into a finite set of reusable phonological components (onsets, vowels, codas), where each component maps to a small set of valid Latin spellings, such that whole-word romanizations can be generated via Cartesian product of component variants?

## Motivation

During the v0.1.0 benchmark labelling process (pipeline/benchmark-v1), three problems emerged with the current rule-based variant generator (`src/variant_generator.py`):

1. **Some rules don't match native speaker behavior.** R-dropping, `ch->c/j` simplification, and blanket `a->aa` lengthening produce variants that Thai speakers don't actually type. The rules were derived from linguistic theory (Research 002) rather than empirical typing patterns.

2. **Labelling is combinatorial.** A word like "เครื่องดื่ม" decomposes into syllables, each into onset+vowel+coda. Labelling at the component level (e.g., กร -> {kr, gr}) is far more efficient than enumerating all whole-word combinations, because components are reusable across words.

3. **Labelling is inconsistent without a reference.** Without a defined component dictionary, the same sound gets transliterated differently across words by the same labeller. A component dictionary eliminates this by defining each mapping once.

## Approach

### Data Source

Use the top 1000-3000 most frequent Thai words extracted from the three primary corpora (wisesight, wongnai, prachathai). The word frequency list is generated by the existing pipeline:

```bash
python -m pipelines.benchmark-wordconv.01_extract_frequencies --top-k 3000
```

Output: `pipelines/benchmark-wordconv/output/word_frequencies.csv`

### Stage 1: Component Inventory

**Goal:** Enumerate all distinct phonological components that appear in the target vocabulary.

**Method:**
1. Run TLTK on the top 1000 words to get syllable decompositions (using the existing `analyze_word()` function from `src/variant_generator.py`)
2. Extract all distinct (initial_cluster, vowel_nucleus, final_consonant) triples
3. Compile a deduplicated inventory of unique onsets, vowels, and codas
4. Cross-reference against Thai phonological references to check for completeness
5. Note any components that appear in theory but not in the top 1000 words

**Expected outcome:** A manageable inventory, likely in the range of:
- 30-50 distinct onsets (initial consonants and clusters)
- 20-30 distinct vowels (simple + compound)
- 10-15 distinct codas (final consonants)
- ~80-100 total component entries to validate

### Stage 2: Draft Component Dictionary

**Goal:** For each component in the inventory, define the set of valid informal Latin romanizations.

**Method:**
1. Start with the RTGS romanization as the base form for each component
2. Add informal variants based on:
   - Native speaker knowledge (maintainer validation)
   - Patterns observed in the v0.1.0 benchmark labels
   - The existing variant generator rules (as a starting reference, not as ground truth)
3. For each variant, document the rationale (e.g., "phonetic simplification", "common Thai internet spelling")
4. Identify components that resist clean decomposition (e.g., vowels that change form based on context, loanword phonemes)

**Dictionary format (draft schema):**

```yaml
# data/dictionaries/component-romanization.yaml
metadata:
  version: "0.1.0"
  description: "Component-level Thai romanization mappings"
  source_vocabulary: "top 1000 words from wisesight+wongnai+prachathai"

onsets:
  ก:      # Thai character(s)
    rtgs: "k"
    variants: ["k", "g"]
    notes: "g voicing common in informal Thai"
  กร:
    rtgs: "kr"
    variants: ["kr", "gr"]
    notes: ""
  ค:
    rtgs: "kh"
    variants: ["kh", "k"]
    notes: "aspiration often dropped in informal spelling"
  # ... etc

vowels:
  ี:       # สระอี (long)
    rtgs: "i"
    variants: ["i", "ee", "ii"]
    notes: "ee most common informal form"
  ิ:       # สระอิ (short)
    rtgs: "i"
    variants: ["i"]
    notes: "short vowel, no lengthened variant"
  # ... etc

codas:
  ด:
    rtgs: "t"
    variants: ["t", "d"]
    notes: "voicing common in informal spelling"
  # ... etc
```

**Key design decisions to resolve during this stage:**
- Should the dictionary key on Thai characters, phonemes, or RTGS romanization?
- How to handle vowels that change written form based on context (สระเปลี่ยนรูป)?
- Should onset clusters (e.g., กร) be separate entries or composed from single consonants?

### Stage 3: Validation Against Benchmark

**Goal:** Measure whether the component dictionary can reproduce known-good romanizations.

**Method:**
1. Write a prototype script that:
   - Decomposes each word into syllables (via TLTK)
   - Looks up each component in the dictionary
   - Generates all romanization variants via Cartesian product
2. Run against the v0.1.0 benchmark (1311 entries, ~300 unique words)
3. Measure:
   - **Component coverage:** What % of components found in the benchmark words are present in the dictionary?
   - **Benchmark reproduction rate:** What % of benchmark `latin_input` entries appear in the generated variants?
   - **Noise ratio:** What fraction of generated variants are NOT in the benchmark? (Lower is better, but some "noise" is expected since the benchmark doesn't list all valid variants)
4. Catalog all failures — words/components the dictionary cannot handle

**Success criteria (defined before experimentation):**
- Component coverage >= 95% (nearly all components in the benchmark vocabulary have dictionary entries)
- Benchmark reproduction rate >= 90% (the dictionary reproduces at least as many benchmark entries as the current rule-based generator)
- The dictionary has <= 120 total entries (manageable size for human validation)
- All dictionary entries are validated by the maintainer as a native speaker

### Stage 4: Summary

Write findings to `research/003-component-romanization/summary.md`:
- The validated component dictionary (or a reference to it in `data/dictionaries/`)
- Coverage metrics against the benchmark
- Edge case catalog
- Recommendations for the next phase (generative validation at scale)

## Out of Scope

- **Loanwords** that use English source spellings (e.g., "taxi", "computer"). Research 002 identified these as needing a separate lookup approach.
- **Frequency/confidence weights** per variant. The dictionary is binary (valid or not). Weighting is a trie-construction concern.
- **Production generator code.** The prototype stays in `experiments/`. Updating `src/variant_generator.py` is deferred to a dedicated pipeline phase (see Change Plan 03 Phase 2).
- **Large-scale generation (10K+ words).** The research validates the dictionary on the existing benchmark. Stress-testing at scale is Phase 2.

## Dependencies

- TLTK (for syllable decomposition and base romanization)
- PyThaiNLP (for corpus tokenization in the word frequency pipeline)
- Existing pipeline: `pipelines/benchmark-wordconv/01_extract_frequencies.py`
- Existing code: `src/variant_generator.py` (for `analyze_word()`)
- Existing benchmark: `benchmarks/word-conversion/v0.1.0.csv`

## Open Questions

- What is the right dictionary key — Thai characters, IPA phonemes, or RTGS strings? Thai characters are the most readable but may be ambiguous (same character, different pronunciation in different positions). RTGS strings are unambiguous but lose the Thai-script connection.
- Should compound vowels (e.g., เ-ือ) be treated as atomic entries or decomposed further?
- At what point during Stage 3 validation do we expand the source vocabulary beyond 1000 words (to 2000 or 3000) to test generalization?
